## 2. NAudio 播放链测试 ❌ 发现根本原因

### 2.1 日志中的关键错误

```
20:37:50.130 [Music] _PlayNextBgm: bgm2 = 7005299 bytes
20:37:50.130 [Music] Stream created, len=7005299
20:37:50.194 [Music] ERROR in _PlayNextBgm:
  FileNotFoundException: 未能加载文件或程序集
  "NAudio.WinMM, Version=2.3.0.0, Culture=neutral, PublicKeyToken=e279aa5131008a41"
  或它的某一个依赖项。系统找不到指定的文件。
```

### 2.2 根因分析

```
Mp3FileReader(Stream)          ← MusicManager.cs 第 159 行
  └─ 依赖 AcmMp3FrameDecompressor  ← NAudio.WinMM.dll
       └─ 依赖 ACM (Audio Compression Manager)  ← Windows 原生 API
```

`Mp3FileReader` 在 NAudio 2.x 架构中位于主 `NAudio.dll`（7KB 门面程序集），但它**运行时依赖** `NAudio.WinMM.dll` 中的 `AcmMp3FrameDecompressor` 来解码 MP3。

项目 `packages.config` 仅安装了 4 个 NAudio 包：
- NAudio 2.3.0 ✅
- NAudio.Core 2.3.0 ✅
- NAudio.Midi 2.3.0 ✅
- NAudio.Wasapi 2.3.0 ✅
- **NAudio.WinMM 2.3.0 ❌ 缺失！**

### 2.3 `DirectSoundOut` 可用性

确认 `DirectSoundOut` 类型存在于 `NAudio.Core 2.3.0`（从 XML 文档验证），该类可用。

### 2.4 修复

已下载并安装 `NAudio.WinMM 2.3.0` NuGet 包到 `packages/`：

| 文件 | 大小 | 位置 |
|------|------|------|
| `NAudio.WinMM.dll` | 57,856 bytes | `lib/netstandard2.0/` |

已更新：
- `packages.config` — 添加 `<package id="NAudio.WinMM" version="2.3.0">`
- `DualMystery.csproj` — 添加 `<Reference Include="NAudio.WinMM">`

**重新编译：成功（0 错误）**。

## 3. 播放器生命周期检查 ⚠️ 基本正常，有小问题

### 3.1 静态字段 GC 风险

`MusicManager` 所有播放器字段均为 `static`，不会被 GC 回收 — **无风险**。

### 3.2 创建/销毁时序

BGM 切换流程：
```
_PlayNextBgm()
  → _StopBgmInternal()     // 停止旧播放器、解绑事件、Dispose
  → new MemoryStream()     // 创建新流
  → new Mp3FileReader()    // 创建新读取器
  → new DirectSoundOut()   // 创建新播放器
  → Init() → Play()        // 启动播放
```

每次切换**正确创建新实例**，不复用已停止的实例。✅

### 3.3 小问题：错误后状态不一致

`_PlayNextBgm()` 中 catch 块调用 `_StopBgmInternal()`，它会将 `_bgmPlayer` 置 null，但 `_bgmStopped` 保持 `false`。

影响：下一次 `StartBgm()` 调用时检查 `_bgmPlayer != null && !_bgmStopped` → `_bgmPlayer` 为 null → 条件为 false → 正确进入重新播放。**实际无功能影响**，但语义不一致。建议 catch 内也设置 `_bgmStopped = true`。

## 4. 线程模型检查 ✅ 正确

| 操作 | 执行线程 | 正确性 |
|------|----------|--------|
| `DirectSoundOut` 创建 | UI 线程（通过 `_dispatchTimer.Tick`） | ✅ |
| `DirectSoundOut.Init()` | UI 线程 | ✅ |
| `DirectSoundOut.Play()` | UI 线程 | ✅ |
| `PlaybackStopped` 回调 | NAudio 内部线程 | ✅ 仅设置标志 |
| 下一曲播放器创建 | UI 线程（Timer 拾取标志） | ✅ |

关键机制：`PlaybackStopped` 回调在 NAudio 内部线程触发，但**仅设置 `_pendingBgmNext = true` 标志**，不直接创建 COM 对象。UI 线程的 `_dispatchTimer`（200ms 间隔）拾取标志后安全创建 `DirectSoundOut`。

**结论**：线程模型设计正确，`DirectSoundOut` 的 COM 线程亲和性得到满足。

## 5. BGM 循环逻辑检查 ✅ 正确

```
bgm1 放完 → PlaybackStopped → flag _pendingBgmNext
  → dispatch timer → _PlayNextBgm() → 清理 + 创建新实例 → 播放 bgm2
bgm2 放完 → PlaybackStopped → flag _pendingBgmNext
  → dispatch timer → _PlayNextBgm() → 清理 + 创建新实例 → 播放 bgm1
...无限循环
```

- 每次切换创建**全新** `DirectSoundOut`，不复用已停止实例 ✅
- `_playBgm1` 布尔值在 `_PlayNextBgm` 中切换 ✅
- `_bgmNaturalEnd` 标志区分"自然结束"和"手动停止" ✅
- `StopBgm()` 设置 `_bgmNaturalEnd = false` 阻止自动循环 ✅

## 6. 结局音乐过渡检查 ✅ 正确

```
PlayEndingMusic()
  → StopBgm()             // 停止 BGM，_bgmStopped=true，_bgmNaturalEnd=false
  → _PlayTruth()          // 创建独立的 _endingPlayer
    → 播放 truth.mp3
    → PlaybackStopped
      → 清理 _endingPlayer/Reader/Stream
      → flag _pendingConanTheme
  → dispatch timer 拾取
    → _PlayConanTheme()   // 创建新的 _endingPlayer
      → 播放 conan_theme.mp3
      → PlaybackStopped → 清理
```

- BGM 和结局音乐使用**独立**的播放器字段（`_bgmPlayer` vs `_endingPlayer`）— 无资源冲突 ✅
- 过渡前正确停止 BGM ✅
- truth → conan_theme 衔接通过 dispatch timer 在 UI 线程安全创建 ✅

### 建议改进（非阻塞）

| 建议 | 优先级 | 说明 |
|------|--------|------|
| 错误后设置 `_bgmStopped = true` | 🟡 低 | 保持状态语义一致 |
| 播放失败后自动重试（延迟 2s） | 🟡 低 | 提高鲁棒性，如音频设备暂时不可用时 |
| 考虑用 `WaveOut` 替代 `DirectSoundOut` | 🟢 建议 | `WaveOut` (即 NAudio 2.x 的 `WaveOutEvent` 替代) 更现代、更可靠，不依赖 DirectSound 遗留 COM 层 |
| 添加 `conan_theme` 资源加载验证 | 🟢 建议 | 当前未到达该代码路径，但字节数确认正常 |
